# W2 系统现状与重构审阅说明（供外部专家）

日期：2026-07-19

代码基线：`600ddc9fa244ca0014bb82248708b53597a592ab`
环境：staging；本说明不包含密钥、数据库、ledger 导出或 provider 凭据。

## 一句话结论

W2 的数据审计、赛果结算和 Dashboard 读取基础已经具备，但产品层把“已部署的基础设施”“仅 staging 验收的能力”“真正可正式推荐的能力”混在一起展示。下一步不应继续堆因子或改文案，而应先收敛为一套可解释的推荐契约，再决定正式推荐是否、何时开放。

## 产品目标与用户真正需要的结果

用户不需要看内部审计代码、历史恢复、技术状态或机器字段。用户需要：

1. 一眼知道今天有哪些比赛值得看、能否正式推荐、推荐什么、何时复核。
2. 只有证据链、盘口链、推荐身份链和结算链完整时，赛后表现才进入命中率与联赛表现。
3. 命中、未中、走水、CLV、联赛表现使用同一批 canonical 样本，数字不能互相矛盾。
4. “数据暂缺”“模型没有优势”“功能尚未开放”“系统异常”必须明确区分。
5. Dashboard 面向老板简洁；分析师和运维才查看证据、诊断和审计细节。

## 当前已经可靠或基本可用的部分

### 赛后验证与统计

- 赛后统计使用 `performance_cohort`：已处理、纳入、排除、待处理分开计算。
- 命中率分母只包含命中与未中；走水不进入分母。
- CLV 只计算 canonical 且满足临场窗口的样本，缺失值不补零。
- 历史身份链缺失的比赛保持审计隔离，不再混入正式统计。
- staging 当前主摘要可展示 canonical 的命中、未中、走水和命中率。

### 数据与运行安全

- Dashboard 的公开读取不调用 provider、不写数据库或 ledger。
- 盘口、推荐、结算与赛果都有身份/新鲜度/完整性检查。
- staging 当前 API 与 Web 代码版本均为 `600ddc9`，数据库健康。

### 分析级多市场能力

- 让球（AH）与大小球（OU）在 `ANALYSIS_PICK` 层独立评分并可择优，不再因数组顺序固定优先让球。
- 首发身份、历史常用阵容基线、阵型/替换/连续性证据和联赛覆盖门禁已经部署到 staging。
- 首发变化可触发重评，五大联赛可以执行严格首发完整性门禁。

## 当前事实：能力成熟度并不一致

| 能力 | 当前状态 | 对用户的意义 |
|---|---|---|
| 赛程、早盘、赛果与验证统计 | 可用 | 可展示，但早盘不是可执行推荐价 |
| AH/OU 分析级推荐 | staging 可用 | 可输出 `ANALYSIS_PICK`，但仍是分析参考 |
| AH 正式 `RECOMMEND` | 代码存在但开关关闭 | 当前不会公开发布正式推荐 |
| OU 正式 `RECOMMEND` | 未完成 | 正式推荐引擎目前只支持 canonical AH |
| 首发证据采集和门禁 | 已部署 | 到临场窗口才会请求官方首发 |
| 首发数值影响 AH/OU | 暂停 | 当前覆盖等级 C，数值调整固定为 0，未证明增量价值 |
| 首发作为正式独立推荐因子 | 未完成 | 不能当作第三个正式独立信号来凑门槛 |
| 生产正式推荐/锁单 | 未开放 | 不应让 UI 暗示已经可以正式下注 |

## 已知问题（按优先级）

### P0：推荐状态的定义不唯一

同一个页面同时出现 `WATCH`、`ANALYSIS_PICK`、`RECOMMEND`、`lock_eligible`、`formal_eligible`、`formal_enabled`、`lineup gate eligible` 等状态。它们来自不同模块，含义不同，却在 UI 中被压缩为“观察/部分就绪”。

后果：用户无法分辨究竟是缺数据、无优势、策略禁用还是系统故障；团队也容易把 staging 通过误说成“功能完成”。

需要：一份唯一的 Recommendation Decision Contract。每场比赛必须只返回一个对用户可见的结论，外加结构化原因：

- `NOT_READY`：缺什么、预计何时补齐；
- `NO_EDGE`：数据完整，但没有独立优势；
- `ANALYSIS_PICK`：分析级建议，说明为什么还不是正式建议；
- `FORMAL_RECOMMEND`：仅在正式路径真正开放且全部门槛通过时出现；
- `SYSTEM_DEGRADED`：数据链或版本链异常，不得伪装成观察。

### P0：让球与大小球在正式路径不对称

AH 和 OU 已能独立进入分析级选择；但正式推荐模块只读取 `canonical_ah_market`。因此 OU 目前永远不能成为 `FORMAL_RECOMMEND`。

需要专家裁决产品范围：

1. 正式推荐短期只做 AH，则 UI、文案和指标必须明确写“正式推荐仅限让球”；或
2. 设计统一的 Market Candidate Contract，使 AH 和 OU 使用同一套报价身份、新鲜度、概率、EV、不确定性、方向一致性、锁定和结算契约。

在这项裁决前，不能将“大小球推荐完成”描述为完整完成。

### P0：旧因子框架与新首发链路并存

当前主因子引擎接入 F1–F9：盘口变化、庄家分歧、体能、重要性、近期赢盘、交锋、强度状态、身价、真实 xG。旧 `F10_LINEUPS_INJURIES` 已有函数，但没有接入主 `FeatureInputs`/定价计算。

同时，今天新上线的 LMM 首发系统走独立链路：首发身份、阵容基线、变更特征、AH/OU 门禁和 provenance。它不是旧 F10 的简单续写。

后果：页面会显示“首发已做/未做”与“独立因子数量”时容易自相矛盾。

需要：建立版本化 Factor Registry，明确每个因子：是否已接入、数据源、是否独立、是否影响 AH、是否影响 OU、是否只作解释、何时可用于正式推荐。决定废弃旧 F10，或让新 LMM 成为唯一的 F10 并接入统一契约。

### P1：独立信号计数被误解为“总因子数量”

正式 AH 路径要求至少 3 个独立权威来源组；这不是“至少 3 个因子”。当前瑞典超三场只有 `ratings` 与 `xg` 两组。体能数据由 xG 历史代理时不应重复计数；盘口和赛事重要性也不应冒充独立模型证据。

需要：Dashboard 只用人话表达“已验证独立证据 2/3，缺少球队历史/交锋/身价等来源”，不要显示模糊的“部分就绪”。专家需要评估 3 组阈值、因子目录和真正的样本外验证是否合理。

### P1：状态诊断存在真实矛盾

当前 API 可同时出现：

- `MISSING_SCORE_MATRIX`，但 `scoreline_readiness = READY`；
- 数据就绪层显示 `ratings` 缺失，但定价因子中已使用内部 Elo ratings；
- 早盘报价保存且可展示，但 `current_odds = {}`，因此正式路径无法计算。

这不是单纯文案问题，而是多个 read model 对“可用”“当前”“参考”的定义不同。

需要：统一 freshness、availability、calculation readiness 的来源与优先级；同一字段只能有一个权威解释。

### P1：首发项目尚未形成真实模型增量

LMM 基础数据和门禁已部署，但当前联赛覆盖等级 C，`lineup_ah_adjustment` 与 `lineup_totals_adjustment` 均为 0。原因是尚未完成冻结离线样本的增量验证。

需要：把“首发采集完成”和“首发影响模型已启用”分开显示。只有在按时间切分的 log loss、Brier/RPS、校准、AH/OU 分市场和置信区间通过预注册门槛后，才开启数值影响。

### P2：Dashboard 信息架构混合了三类用户

老板需要结论与结果；分析师需要价格、证据和市场变化；运维需要 provider、版本、数据完整性和审计。这三类信息仍在同一页面和同一组件层中互相泄漏。

需要：老板首页、分析详情、技术诊断三层分离；审计入口默认不干扰老板视图。

## 建议的改造顺序

### 第一阶段：先收口契约，不增加新功能

1. 定义单一 `RecommendationDecisionV3` 响应：唯一结论、唯一原因、数据状态、市场状态、模型状态、可见性、审计引用。
2. 明确 AH/OU 的产品范围：要么正式路径统一支持，要么明确限制正式推荐为 AH。
3. 建立 Factor Registry，移除或迁移旧 F10，禁止同一首发信息在两条链路中重复解释。
4. 统一“最新可执行盘口”“历史参考盘口”“已过期盘口”的 schema 和 UI 文案。
5. 以契约测试锁定上述语义，禁止 Dashboard 各组件自行拼状态。

验收：同一 fixture 在 API、Dashboard、锁定快照和赛后统计中看到的市场、选择、状态和原因完全一致。

### 第二阶段：补齐真正的推荐能力

1. 若产品要正式推荐 OU，建立其完整的报价身份、EV、不确定性、结算与 CLV 链。
2. 对 LMM 首发调整做离线消融和前向 shadow；未通过前始终保持数值权重为 0。
3. 建立联赛级数据覆盖表；不同联赛不能默认使用同一首发门槛。
4. 重新评估独立来源阈值及每个因子的样本外贡献，避免用弱相关或代理数据“凑三组”。

### 第三阶段：按角色重做 Dashboard

1. 老板首页只展示：今天可行动比赛、结论、复核时间、关键风险、赛后 canonical 表现。
2. 分析详情展示：AH/OU 候选、最新盘口、市场变化、模型与市场差异、关键证据及失效条件。
3. 技术诊断只在展开页：provider、版本、字段新鲜度、审计 ID、数据链异常。
4. 所有数字都可钻取，但默认不展示“历史恢复”“identity conflict”等内部术语。

## 需要专家给出的决策

1. W2 应该先把 AH 做成唯一可信的正式推荐市场，还是同时完成 AH/OU 正式路径？
2. “正式推荐”的最小证据集是什么？3 个独立来源是否有实证依据？
3. 首发应当是硬门、数值模型特征，还是只作高风险提示？不同联赛如何分级？
4. 如何将市场概率作为基准，同时避免把市场信息重复计入“独立优势”？
5. 哪些指标应留在老板首页，哪些只应出现在分析/运维层？
6. 应怎样设计测试和发布门禁，防止“代码存在”被误说成“能力已可用”？
