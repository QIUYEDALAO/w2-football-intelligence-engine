# W2 Master Roadmap

本文件回答“项目最终要完成什么”。

本文件是 W2 的长期权威路线：总决策、执行顺序、阶段 0–15、Gate 0–6、严格禁止事项和并行实时盘口采集原则均以用户批准的 master plan 为来源。

内部 Stage7I、Stage10E、Patch、Release Train 等名称属于 execution package，不等于本路线中的阶段编号。execution package 可以映射到路线阶段，但不得改写路线阶段。

本文件原则上不随普通执行包重写；只有用户明确批准总范围变更时，才更新本文并递增 roadmap_version。

## 用户批准 Master Plan 原文

总决策
停止继续修补 W1，冻结为 Legacy；新建 W2 Football Intelligence Engine。
这不是把现有成果全部删除，而是：
	•	保留原始数据、盘口快照思想、Dixon-Coles 市场基准、赛前锁定、赛后审计和防数据泄漏规则。
	•	放弃当前以 scripts/、本地 JSON、静态 Dashboard 和超长脚本拼起来的运行架构。
	•	W1 只负责历史参考和临时采集，W2 才是未来世界杯、五大联赛和其他一级联赛共用的生产系统。
原因很明确：
	1	W1 最初被定义为“世界杯专用、数据审计和蓝图设计”，甚至明确禁止生成正式推荐，产品边界与现在的目标完全不同。
	2	当前核心模型由 1X2 和 OU 盘口反解参数，本质是市场概率复述；阵容、战术等没有进入核心概率，因此结构上不具备独立于市场的优势。
	3	当前候选生成器自己也明确标记 independent_edge=false、calibrated=false，它不能被当成正式推荐引擎。
	4	现有历史回测仍是 1X2_ONLY，明确没有验证完整的 OU→μ→λ→比分矩阵管线。
	5	现有独立球队强度原型虽然优于均匀基准，但仍明显不如市场基准，不能接生产。
	6	盘口变化目前最大的阻碍不是算法，而是数据：当前审计显示没有真正开盘赔率，赔率更新约每天一次，API 配额也很有限，这不足以支持严肃的盘口时序分析。

全部执行顺序

冻结 W1
→ 立即保住实时盘口快照
→ 定义产品与验收标准
→ 建立 W2 工程底座
→ 建立统一数据模型
→ 接通数据采集
→ 建立历史 As-of 数据集
→ 建立市场基准
→ 建立独立概率模型
→ 建立完整回测系统
→ 建立比赛筛选与推荐策略
→ 建立 API 和后台
→ 建立监控与部署
→ 世界杯影子运行
→ 正式切换
→ 五大联赛及其他联赛扩展

其中，实时盘口采集必须与其他任务并行立即开始，因为错过的盘口历史无法事后补回来。现有专家报告也把这件事列为必须立即启动的时间敏感任务。

	•	只允许修复影响数据采集的严重故障。
	•	建立 W1_LEGACY_STATUS.md，写明 W1 不再作为未来生产架构。
0.2 保存当前基线
	•	运行现有全部 checker。
	•	保存每个 checker 的结果、运行环境、Python 版本和依赖。
	•	保存当前 Dashboard 输出。
	•	保存当前模型版本、配置文件和阈值。
	•	保存当前世界杯所有比赛卡、结果、赔率和日志。
	•	给所有保存文件生成 SHA256 校验值。
0.3 对 W1 文件分类
每个文件必须归入以下一类：
	•	PORT：迁移思想或代码。
	•	REFERENCE：仅作研究参考。
	•	MIGRATE_DATA：数据迁移。
	•	ARCHIVE：封存。
	•	DELETE_LATER：W2 上线后删除。
0.4 明确保留资产
保留并重写以下思想：
	•	原始盘口不可变存储。
	•	每个 bookmaker 单独保存。
	•	赔率采集时不去水、不做共识、不改写。
	•	赛前预测锁定。
	•	赛果只进入评估层，不自动回流调参。
	•	Dixon-Coles 模型作为“市场基准模型”。
	•	Forward Ledger 的 append-only 思想。
	•	数据泄漏 checker。
	•	API-Football 字段映射经验。
	•	球队和 fixture ID 映射经验。
当前盘口采集器“写入时只保存原始赔率、不去水、不聚合”的设计是正确的，应迁移到 W2。
0.5 明确废弃部分
不迁移以下架构：
	•	w1_local_predict_server.py 式单文件服务器。
	•	以本地 JSON 文件作为任务状态和生产数据库。
	•	模型运行后重新构建整份静态 Dashboard。
	•	模型代码兼容旧 UI 字段的做法。
	•	硬编码世界杯球队中文名、场馆和 fixture。
	•	从其他旧项目目录中读取 API Key。
	•	手工维护大量 check_w1_xxx.py 作为主要测试体系。
	•	scripts/ 目录承载所有业务代码的做法。
当前服务将 API 调用、凭据读取、状态文件、进度管理、首发、天气和 Dashboard 重建放在同一个运行进程中，这是必须拆除的耦合。

阶段 1：确定 W2 产品边界
1.1 产品定位
W2 定义为：
一个多联赛、可回测、可审计、基于赛前数据和盘口时序的足球概率分析与比赛筛选系统。
它不是单纯的“比分预测器”，也不是“盘口变化大就自动推荐”的规则程序。
1.2 第一版支持市场
正式推荐第一阶段只支持：
	1	1X2。
	2	亚洲让球 AH。
	3	大小球 OU。
	4	BTTS 作为第二阶段候选。
精确比分暂时只作为比赛解释和概率分布展示，不作为核心正式推荐市场。
1.3 输出状态统一
每场比赛只能处于以下状态之一：
	•	NOT_READY：数据不足。
	•	SKIP：明确跳过。
	•	WATCH：值得观察，但未达到正式标准。
	•	CANDIDATE：有候选方向，但仍未通过全部风控。
	•	RECOMMEND：通过正式推荐标准。
	•	LOCKED：推荐已锁定，不允许赛后修改。
	•	SETTLED：赛果结算完成。
默认状态必须是 SKIP，不能为了每天有内容而强行推荐。
1.4 预测时间点
统一建立：
	•	T-72h。
	•	T-48h。
	•	T-24h。
	•	T-12h。
	•	T-6h。
	•	T-3h。
	•	T-1h。
	•	T-30m。
	•	T-10m。
	•	Closing。
每次预测必须记录独立的 as_of_time，禁止后续数据污染早期预测。
1.5 成功标准
系统不再以“命中率”作为唯一指标。
模型层指标：
	•	Log Loss。
	•	Ranked Probability Score。
	•	Brier Score。
	•	Calibration Error。
	•	各概率区间可靠性曲线。
	•	对市场基准的增量表现。
策略层指标：
	•	推荐覆盖率。
	•	Closing Line Value。
	•	不同市场、联赛和赔率区间的稳定性。
	•	固定规则下的历史结果。
	•	最大连续错误和风险分布。
	•	推荐后盘口方向是否继续支持。
	•	在不同时间窗口是否稳定。
系统层指标：
	•	数据采集成功率。
	•	数据新鲜度。
	•	任务失败率。
	•	API 配额使用率。
	•	预测生成延迟。
	•	推荐锁定完整率。
	•	赛果同步完整率。

阶段 4：立即建立正式数据采集系统
这是整个项目的第一条关键路径。
4.1 API-Football 适配器
分别实现：
	•	fixtures。
	•	teams。
	•	standings。
	•	odds。
	•	lineups。
	•	injuries。
	•	squads。
	•	fixture detail。
	•	results。
	•	events。
	•	statistics。
每个 endpoint 独立模块，不允许写在同一个服务器文件中。
4.2 赔率数据源决策
API-Football 可以继续作为：
	•	赛程主源。
	•	球队和球员主源。
	•	首发、伤停、赛果主源。
	•	赔率辅助源。
但盘口变化正式系统需要另一个能够提供以下能力的赔率主源：
	•	更高更新频率。
	•	多 bookmaker。
	•	开盘或尽早赔率。
	•	历史快照。
	•	1X2/AH/OU。
	•	closing odds。
	•	稳定事件 ID。
	•	可接受的商业使用条款。
必须建立数据源对比测试，不能仅凭价格选择。
4.3 实时快照调度
按比赛时间自动安排采集任务：
	•	距比赛超过 48 小时：低频。
	•	T-48h 至 T-12h：中频。
	•	T-12h 至 T-3h：提高频率。
	•	T-3h 至 T-1h：高频。
	•	T-1h 至 T-10m：最高允许频率。
	•	开赛后停止赛前盘口采集。
	•	保存 closing snapshot。
4.4 配额管理
建立：
	•	每日配额预算。
	•	endpoint 成本表。
	•	优先级队列。
	•	世界杯比赛优先。
	•	即将开赛比赛优先。
	•	已完成比赛不再消耗赔率请求。
	•	配额不足时自动降频。
	•	API 剩余配额告警。
4.5 稳定性机制
每个采集任务必须具备：
	•	幂等。
	•	重试。
	•	指数退避。
	•	超时。
	•	熔断。
	•	去重。
	•	原始 Payload Hash。
	•	请求日志。
	•	provider response time。
	•	provider error code。
	•	数据延迟记录。
4.6 数据新鲜度告警
告警条件包括：
	•	比赛临近但没有赔率。
	•	赔率超过指定时间未更新。
	•	bookmaker 数突然下降。
	•	盘口市场消失。
	•	fixture 状态冲突。
	•	主客队反转。
	•	开赛时间变更。
	•	lineup 应出现但未出现。
	•	result 长时间不同步。

阶段 5：建立历史训练与回测数据集
5.1 历史数据源
需要建立两套历史数据：
国家队数据集
用于世界杯和国际赛事：
	•	国际友谊赛。
	•	世界杯预选赛。
	•	洲际杯赛。
	•	世界杯。
	•	Nations League 等正式赛事。
	•	赛果。
	•	1X2。
	•	OU。
	•	AH。
	•	中立场。
	•	比赛重要性。
	•	队伍阵容可用性。
俱乐部联赛数据集
用于八月后的联赛扩展：
	•	至少覆盖多个完整赛季。
	•	开盘和收盘赔率。
	•	1X2/AH/OU。
	•	赛果。
	•	主客场。
	•	赛程密度。
	•	升降级。
	•	欧洲赛事干扰。
	•	队伍变更。
当前 W1 的历史数据缺口非常明显，尤其国际比赛 OU/AH 曾有 1081/1081 缺失，完整模型无法依赖现有数据直接验收。
5.2 As-of 数据集
回测数据不能只有“最终结果表”，必须能还原某一时刻系统实际知道什么。
每一行训练样本必须包含：
	•	fixture。
	•	prediction_time。
	•	当时可见的赔率。
	•	当时可见的阵容信息。
	•	当时可见的伤停。
	•	当时可见的球队历史数据。
	•	当时模型版本。
	•	最终结果单独作为 label。
5.3 防泄漏审计
严格禁止：
	•	使用未来赛果生成赛前评分。
	•	使用 closing odds 回测 T-24h 预测。
	•	使用赛后修正过的伤停。
	•	使用比赛后更新的球队排名。
	•	随机切分跨时间数据。
	•	同一场比赛的不同快照跨 train/test。
	•	赛季后半段数据影响赛季前半段预测。
5.4 数据质量测试
每次数据构建必须检查：
	•	唯一性。
	•	缺失率。
	•	数值范围。
	•	主客一致性。
	•	时间顺序。
	•	ID 映射。
	•	赔率倒数和。
	•	盘口 line 解析。
	•	bookmaker 数。
	•	结果状态。
	•	重复比赛。
	•	异常大比分。
	•	中立场标记。
	•	赛季和联赛归属。

阶段 6：建立市场分析层
市场层只负责描述市场，不负责宣称独立优势。
6.1 Bookmaker 共识
建立可配置的 bookmaker 权重：
	•	排除低质量或长期异常 bookmaker。
	•	记录每家盘口更新时间。
	•	使用中位数或稳健加权方式。
	•	计算 bookmaker 分歧。
	•	计算有效 bookmaker 数。
	•	对过旧盘口降权。
	•	不把单一 bookmaker 当市场共识。
6.2 去水方法
同时实现并回测：
	•	Proportional。
	•	Shin。
	•	Power。
	•	Logarithmic。
不同市场和赔率区间可以选择不同方法，但必须由样本外结果决定。
6.3 OU 参数拟合
不再将 OU 的 50% 中位线直接当作总进球均值。
改为：
	•	使用完整 OU 阶梯。
	•	最小化模型与多档 OU 市场概率的整体误差。
	•	输出拟合残差。
	•	拟合失败时降级。
	•	记录不同 line 的市场冲突。
现有报告已经指出中位线直接作为 μ 会产生系统偏差，完整 OU 阶梯拟合是明确的改进方向。
6.4 盘口变化特征
每个市场计算：
	•	首次采集至当前变化。
	•	最近一次变化。
	•	变化速度。
	•	变化加速度。
	•	不同 bookmaker 是否同步。
	•	主线是否变化。
	•	水位变化。
	•	bookmaker 分歧变化。
	•	1X2、AH 和 OU 是否同向。
	•	首发公布前后变化。
	•	新闻事件前后变化。
	•	临近开赛的反向变化。
	•	closing 前最终变化。
6.5 市场质量评分
每场生成：
	•	流动性等级。
	•	bookmaker 覆盖等级。
	•	数据新鲜度。
	•	盘口分歧程度。
	•	是否有异常源。
	•	是否可以正式分析。
	•	是否只能观察。
	•	是否必须跳过。

阶段 7：建立独立概率模型
目标不是重新复制盘口，而是建立独立于盘口的基础概率，再分析两者残差。
7.1 动态球队强度
实现：
	•	时间衰减 Elo。
	•	主客场优势。
	•	中立场调整。
	•	比赛重要性权重。
	•	对手强度。
	•	洲际和联赛强度。
	•	小样本收缩。
	•	新球队或升班马先验。
	•	国家队长期不比赛的衰减。
7.2 进球模型
至少比较：
	•	独立 Poisson。
	•	Dixon-Coles。
	•	双变量 Poisson。
	•	Negative Binomial。
	•	分层攻防模型。
	•	带时间衰减的攻防模型。
世界杯和联赛不得强行使用同一组参数。
7.3 世界杯专用特征
	•	中立场。
	•	东道主。
	•	小组赛或淘汰赛。
	•	小组出线形势。
	•	必须赢、可接受平局等情景。
	•	休息天数。
	•	旅行距离。
	•	海拔。
	•	气温和湿度。
	•	加时可能性与 90 分钟市场区分。
	•	队伍近期国际比赛样本稀疏度。
7.4 联赛专用特征
	•	主场优势。
	•	赛程密度。
	•	欧战影响。
	•	杯赛影响。
	•	长途旅行。
	•	新教练。
	•	升班马。
	•	球队阵容价值变化。
	•	赛季阶段。
	•	保级、争冠和欧战资格压力。
7.5 阵容和伤停模型
阵容不能直接凭主观判断加减概率。
需要：
	•	球员价值评分。
	•	预计上场时间。
	•	替代球员质量。
	•	门将单独建模。
	•	中卫组合。
	•	核心进攻球员。
	•	阵型变化。
	•	首发确认前使用概率阵容。
	•	首发确认后重新计算。
	•	通过历史消融证明增量价值。
7.6 市场残差模型
核心目标：

独立模型概率 - 市场公平概率

残差模型研究：
	•	市场是否高估热门。
	•	是否低估平局。
	•	某些联赛主场优势是否被低估。
	•	某些盘口阶段是否存在稳定偏差。
	•	首发后盘口是否充分吸收信息。
	•	小 bookmaker 变化是否领先主流 bookmaker。
	•	市场分歧是否预示不确定性，而非机会。
7.7 概率校准
对最终输出进行：
	•	Isotonic。
	•	Platt。
	•	Beta calibration。
	•	多分类 Dirichlet calibration。
	•	分联赛校准。
	•	分市场校准。
	•	分赔率区间校准。
	•	分比赛阶段校准。
输出必须包含不确定性区间，而不是只给一个点概率。

阶段 8：建立完整回测系统
8.1 事件驱动 Replay
回测引擎必须按历史时间顺序重放：
	1	读取当时可用数据。
	2	构建当时特征。
	3	加载当时模型版本。
	4	生成概率。
	5	运行推荐策略。
	6	锁定输出。
	7	比赛结束后结算。
	8	汇总指标。
8.2 时间切分
采用：
	•	Rolling window。
	•	Expanding window。
	•	Walk-forward。
	•	严格未来测试集。
	•	按赛季和比赛日期切分。
8.3 基准对比
所有模型必须和以下基准比较：
	•	均匀概率。
	•	纯 Elo。
	•	纯市场去水概率。
	•	现有 W1 Dixon-Coles 市场模型。
	•	简单 Poisson。
	•	上一生产模型版本。
8.4 模型指标
至少输出：
	•	Log Loss。
	•	RPS。
	•	Brier。
	•	ECE。
	•	Reliability Plot。
	•	1X2 分项表现。
	•	OU 分项表现。
	•	AH 分项表现。
	•	比分分布 Log Score。
	•	对市场基准的差值和置信区间。
8.5 策略指标
	•	候选数量。
	•	正式推荐数量。
	•	推荐覆盖率。
	•	平均市场赔率。
	•	平均模型优势。
	•	Closing Line Value。
	•	固定单位条件下的历史结果。
	•	分联赛表现。
	•	分 bookmaker 表现。
	•	分预测时间表现。
	•	分盘口区间表现。
	•	连续错误分布。
	•	策略对阈值变化的敏感性。
8.6 消融实验
分别移除：
	•	盘口变化。
	•	阵容。
	•	Elo。
	•	天气。
	•	旅行。
	•	休息天数。
	•	比赛重要性。
	•	市场残差模型。
	•	校准层。
只有能在样本外稳定提升的特征才能进入生产。
8.7 模型晋级标准
模型进入生产至少需要：
	•	样本外 Log Loss 或 RPS 优于当前基准。
	•	改善不是由单一联赛或极少数比赛贡献。
	•	Calibration 不恶化。
	•	时间切分后仍稳定。
	•	多个市场区间一致。
	•	bootstrap 置信区间支持增量。
	•	无数据泄漏。
	•	能完全重现。
	•	通过 shadow run。
若独立模型不能证明优于市场，系统可以继续做市场分析，但正式状态必须保持 WATCH 或 SKIP，不能伪装成有优势的推荐。

阶段 9：建立比赛筛选和推荐引擎
9.1 候选生成
候选只能来自真实可用盘口：
	•	市场当前确实有该 line。
	•	至少达到最低 bookmaker 数。
	•	赔率足够新鲜。
	•	不是暂停盘口。
	•	不是比赛已经开始后的 live 盘口。
	•	settlement 规则明确。
9.2 公平赔率计算
每个候选输出：
	•	独立模型概率。
	•	校准后概率。
	•	市场公平概率。
	•	当前可得赔率。
	•	公平赔率。
	•	模型优势。
	•	不确定性区间。
	•	bookmaker 数。
	•	数据时间。
	•	模型版本。
9.3 推荐硬门槛
出现以下情况直接 SKIP：
	•	核心盘口缺失。
	•	赔率过旧。
	•	bookmaker 数不足。
	•	阵容状态异常。
	•	fixture 映射冲突。
	•	模型数据缺失。
	•	模型间严重分歧。
	•	校准区间样本不足。
	•	盘口变化来源不可靠。
	•	模型优势低于不确定性。
	•	该联赛或市场尚未通过生产验收。
9.4 推荐分层
	•	A：正式推荐，数据和模型全部通过。
	•	B：候选，存在一定优势但有警告。
	•	C：观察，不构成正式输出。
	•	SKIP：不参与。
A 级不能靠人工强制提升。
9.5 相关性控制
同一场比赛不得无节制推荐多个高度相关市场，例如：
	•	主胜与主队 -0.5。
	•	大球与双方进球。
	•	强队大胜与热门让深盘。
系统需要：
	•	计算候选相关性。
	•	选一个主要候选。
	•	最多保留一个低相关备选。
	•	防止同一信息重复计数。
9.6 推荐解释
每条推荐必须给出：
	•	为什么进入候选。
	•	模型与市场差异来自哪里。
	•	盘口变化是否支持。
	•	哪些因素反对。
	•	最大风险。
	•	数据缺口。
	•	什么变化会让推荐失效。
	•	推荐锁定时间。
	•	使用的模型和策略版本。
9.7 不可变推荐账本
推荐一旦锁定：
	•	不允许覆盖。
	•	不允许赛后修改概率。
	•	不允许修改推荐方向。
	•	只能追加新版本或撤销事件。
	•	每次修改必须保存操作者、原因和时间。
	•	结果同步只能进入 settlement 和 evaluation。

	1	禁止继续在 W1 主架构上叠加新功能。

	12	禁止为了每天有推荐而降低门槛。
